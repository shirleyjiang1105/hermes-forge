# 文献调研交互式SOP

## 触发流程

当用户表明要写SCI/项目申请/发明专利时，按以下步骤逐步引导。**每步完成后再进入下一步，不要跳过。**

---

## Step 0：识别文档类型 + 确认核心问题

询问用户：
1. 文档类型：SCI论文 / 项目申请(国自然/教育部/上海市) / 发明专利
2. 核心研究问题（一句话）
3. 目标期刊或申报口（如果知道）

确认后，将信息回显给用户确认，然后进入对应类型的SOP。

---

## Step 1：定义PICOS/研究框架

根据文档类型使用不同框架：

**SCI论文 → PICOS框架**
```
P (Population): 研究对象是谁？
I (Intervention): 核心干预/暴露是什么？
C (Comparison): 跟什么比？
O (Outcome): 看什么结果？
S (Study design): 什么设计？
```

**项目申请 → 四段式**
```
政策需求：国家/部委/省市要什么？（引用具体文号）
实践需求：教学/临床一线缺什么？
学术空白：文献里没解决什么？
你的方案：你打算怎么做？
```

**发明专利 → 技术对比**
```
现有技术方案A：手段+缺点
现有技术方案B：手段+缺点
本发明方案：手段+解决了什么缺点
```

输出：将框架填入Markdown文件保存到 `~/research-mgmt/lit-review/{日期}-{项目简称}/`

---

## Step 2：多库检索

执行检索（按以下优先级，根据可访问性调整）：

**必做**：
- PubMed (Entrez API，免费)：2-3组关键词
- CNKI (如果有学校VPN)

**选做**：
- WoS/Scopus (学校VPN)
- IEEE Xplore (工程/计算机方向)
- Google Scholar (手动)

**检索策略**：
- 核心检索：主题词组合（如"portable EEG AND cognitive load AND education"）
- 补充检索：关键作者追踪
- 反向检索：核心文献的参考文献和被引

检索结果导出为RIS或格式化文本，保存到项目目录。

**工具**：使用 `/root/pubmed_search.py` 的增强版脚本进行PubMed检索。

---

## Step 3：三级筛选

引导用户逐轮筛选（如果用户参与筛选；如果用户说"你帮我筛"，则Agent自行判断）：

1. **标题筛选**：排除明显不相关 → 保留约50%
2. **摘要筛选**：排除方法/人群/场景不匹配 → 保留15-25篇
3. **全文筛选**：精读后保留核心10-15篇

输出筛选记录（Markdown表格：标题/作者/年份/期刊/DOI/入选理由/排除理由）。

---

## Step 4：画文献地图

生成三轴地图：

**时间轴**：核心文献按年份排列，标注方法演进节点
**方法轴**：按方法层级分类（实验室→半自然→真实场景）
**结论轴**：标注共识（✓）与争议（⚡）

输出为文本图表（ASCII或Markdown）。

---

## Step 5：找缺口

对照五种缺口类型逐一排查：

| 类型 | 说明 | 含金量 |
|------|------|--------|
| 矛盾 | A说X，B说非X | ⭐⭐⭐⭐⭐ |
| 方法迁移 | 方法M在B领域验证了，A领域没做过 | ⭐⭐⭐⭐ |
| 人群空白 | 人群P1做过，P2没做过 | ⭐⭐⭐ |
| 生态效度 | 实验室验证了，真实场景没验证 | ⭐⭐⭐⭐⭐ |
| 机制空白 | 知道现象，不知道为什么 | ⭐⭐⭐⭐⭐ |

输出一句**缺口陈述**："现有文献在X方面的共识是Y，但在Z点上存在分歧/空白。"

---

## Step 6：写背景段

根据文档类型输出对应模板的背景段草稿：

**SCI论文 → 漏斗式**
> 大背景(2句) → 现有进展(3-4句,引5-8篇) → 存在的缺口(1-2句) → 本研究的回答(1句) → 假设(1句)

**项目申请 → 政策-实践-学术-方案**
> 政策需求 → 实践需求 → 学术空白 → 你的方案

**发明专利 → 现有技术-本发明-对比**
> 现有技术方案及缺点 → 本发明方案 → 逐项对比

输出可直接使用的文本。

---

## Step 7：反向检查

1. □ 正文引用是否与参考文献一一对应？
2. □ 是否有反例未讨论？（刻意找一篇结论相反或null result的文献）
3. □ 关键词是否覆盖了所有相关子方向？
4. □ 缩写是否首次出现时定义了全称？

---

## 输出清单

每次完成后生成以下文件到 `~/research-mgmt/lit-review/{日期}-{项目简称}/`：

```
{日期}-{项目简称}/
├── 01-PICOS.md          # 研究框架定义
├── 02-search-results.md  # 检索结果汇总
├── 03-screening.md       # 筛选记录
├── 04-literature-map.md  # 文献地图
├── 05-gap-analysis.md    # 缺口分析
├── 06-background-draft.md # 背景段草稿
├── 07-checklist.md       # 反向检查清单
└── raw/                  # 原始检索结果
```

---

## 注意事项

- **每步完成后询问用户是否满意再继续**（除非用户明确说"全自动"）
- **写作铁律**：绝不编造文献DOI；缩写首次出现必定义；引用与参考文献一一对应
- 如果用户已有Zotero库，询问是否从Zotero导入已有文献
- 检索到的文献自动记录DOI，未找到DOI的标注"待验证"
